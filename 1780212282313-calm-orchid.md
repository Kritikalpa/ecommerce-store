# Ecommerce Store — Fullstack Monorepo Plan

## Overview

A monorepo containing a **Node.js/TypeScript REST API backend** and a **React/TypeScript frontend**, implementing cart management, checkout with discount codes, and an admin dashboard. All state is held in-memory (no database).

---

## Tech Stack

| Layer | Choice | Rationale |
|---|---|---|
| Monorepo tooling | pnpm workspaces + Turborepo | Fast, cache-aware builds; native workspace linking |
| Backend | Node.js + TypeScript + Fastify | Type-safe, fast HTTP framework with schema validation |
| Frontend | React 18 + TypeScript + Vite | Best DX, fast HMR, native ESM |
| Styling | TailwindCSS | Utility-first, no runtime overhead |
| State (FE) | Zustand | Lightweight, no boilerplate vs. Redux |
| API client | Axios | Interceptors for session-id header injection |
| Shared types | `packages/shared` | Single source of truth for request/response types |
| Testing | Vitest + Supertest | Compatible with ESM, fast, Jest-compatible API |
| Linting | ESLint + Prettier | Consistent code quality |

## Data Models (in `packages/shared`)

```typescript
// Product
interface Product {
  id: string;
  name: string;
  description: string;
  price: number;          // in cents to avoid floating-point issues
  imageUrl: string;
  stock: number;
}

// Cart
interface CartItem {
  productId: string;
  quantity: number;
  unitPrice: number;      // price snapshot at time of add
}
interface Cart {
  sessionId: string;
  items: CartItem[];
  updatedAt: string;      // ISO date
}

// Order
interface OrderItem extends CartItem {
  productName: string;
}
interface Order {
  id: string;
  sessionId: string;
  items: OrderItem[];
  subtotal: number;
  discountCode?: string;
  discountAmount: number;
  total: number;
  createdAt: string;
}

// Discount
interface DiscountCode {
  code: string;
  discountPercentage: number;
  isUsed: boolean;
  createdForOrderCount: number;   // which nth order triggered generation
  usedInOrderId?: string;
  createdAt: string;
}

// In-memory store shape (backend only)
interface StoreState {
  products: Map<string, Product>;
  carts: Map<string, Cart>;       // keyed by sessionId
  orders: Order[];
  discountCodes: Map<string, DiscountCode>;
  pendingDiscountGeneration: number; // how many codes can still be generated
}
```

## API Design

### Customer APIs

| Method | Path | Description |
|---|---|---|
| GET | `/api/products` | List all products |
| GET | `/api/products/:id` | Get a product |
| GET | `/api/cart` | Get current session cart |
| POST | `/api/cart/items` | Add item to cart |
| PUT | `/api/cart/items/:productId` | Update item quantity |
| DELETE | `/api/cart/items/:productId` | Remove item from cart |
| DELETE | `/api/cart` | Clear cart |
| POST | `/api/orders/checkout` | Checkout (apply discount code) |
| GET | `/api/orders/:id` | Get order details |

**Header:** All customer requests include `x-session-id: <uuid>` (auto-injected by frontend).

#### POST `/api/cart/items`
```json
Request:  { "productId": "p1", "quantity": 2 }
Response: Cart
```

#### POST `/api/orders/checkout`
```json
Request:  { "discountCode": "SAVE10-XXXX" } // optional
Response: Order
```

### Admin APIs

| Method | Path | Description |
|---|---|---|
| POST | `/api/admin/discount-codes/generate` | Generate a discount code if nth-order condition is met |
| GET | `/api/admin/stats` | Aggregate store statistics |

**Auth:** `x-admin-key: <ADMIN_API_KEY>` header required.

#### POST `/api/admin/discount-codes/generate`
```json
Response (success): { "code": "SAVE10-A1B2", "discountPercentage": 10 }
Response (failure): 400 { "error": "No pending discount codes to generate" }
```

#### GET `/api/admin/stats`
```json
Response:
{
  "totalOrders": 12,
  "totalItemsPurchased": 47,
  "totalRevenue": 89400,         // in cents
  "totalDiscountGiven": 4200,    // in cents
  "discountCodes": [
    { "code": "SAVE10-A1B2", "isUsed": true, "usedInOrderId": "o-5" },
    { "code": "SAVE10-C3D4", "isUsed": false }
  ]
}
```

## Core Business Logic

### Discount System

- Config: `DISCOUNT_EVERY_NTH_ORDER=5`, `DISCOUNT_PERCENTAGE=10` (env vars with defaults)
- **On checkout:** When an order is successfully placed and `orders.length % N === 0`, increment `pendingDiscountGeneration` by 1.
- **Admin generate:** If `pendingDiscountGeneration > 0`, create a new discount code, decrement counter.
- **Discount code format:** `SAVE{X}-{RANDOM_4_CHARS}` e.g. `SAVE10-A1B2`
- **Validation on checkout:** Code exists, is not already used, belongs to `discountCodes` map.
- **One-time use (global):** After successful checkout with a code, mark `isUsed = true`. No other customer can use it again.

### Cart Service
- Cart is keyed by `sessionId`
- Adding an existing product increases quantity
- Removing an item that doesn't exist is a no-op (idempotent)
- Cart is cleared after successful checkout

### Checkout Service
1. Validate cart is non-empty
2. Validate discount code if provided
3. Calculate subtotal from cart items (price snapshots)
4. Apply discount if code valid
5. Create Order record
6. Clear cart
7. Check and increment `pendingDiscountGeneration` if nth-order threshold hit
8. Return Order

## Session Management

- On app load, check `localStorage` for `ecommerce_session_id`
- If absent, generate a `crypto.randomUUID()` and persist it
- All API requests include `x-session-id` header via Axios interceptor
- Backend creates a cart tied to `sessionId` on first cart operation

## DECISIONS.md (5+ Decisions to Document)

1. **Fastify over Express** — Schema-based validation, better TypeScript support, faster performance
2. **pnpm workspaces + Turborepo** — Efficient monorepo with shared `packages/shared` for types; cache-aware task runner
3. **In-memory singleton store** — Simple Map-based store with a single instance; thread safety not a concern in single-process Node.js
4. **Price stored in cents (integer)** — Avoids floating-point arithmetic errors in financial calculations
5. **Session-based cart (no auth)** — UUID in localStorage for anonymous users; simple and sufficient without user registration
6. **Admin-triggered discount code generation** — Instead of auto-generating on checkout, admin explicitly calls the generate API when the condition is met; this gives control and mimics a real business workflow
